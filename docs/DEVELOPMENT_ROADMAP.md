# Development Roadmap

## Purpose

Complete and harden the backend execution system in a staged, results-first way, and only afterward move into structural review and frontend-aligned review.

This roadmap exists to prevent unproductive loops of:
- analysis
- notes
- backlog-only iterations

From this point forward, each stage must produce tangible progress.

---

## Working Principles

### 1. Every stage must produce real results

A valid result is one of these:
- functional production code
- a corrected behavior
- a test executed with real result
- a missing module implemented

Documentation is required, but it does not replace execution.

### 2. Do not mix change types unnecessarily

Each stage should focus on one main type of work:
- robustness
- coverage
- refactor
- integration

### 3. Close stages before opening new ones

A stage is only considered closed when it has:
- implementation or real executed test result
- minimal validation
- minimal documentation
- manual commit
- clean tree

### 4. Documentation follows execution

Notes, logs, and backlog entries are secondary to the actual deliverable.
A documentation-only iteration is not considered sufficient progress unless it closes a prior executed change.

---

## Stage 1 — Backend Robustness (Broker-facing)

### Goal

Implement the missing backend capabilities that make the execution system more robust and professional in real broker interaction scenarios.

This stage is focused on actual execution reliability, not cosmetic restructuring.

---

### 1.1 Stable client_order_id toward broker

#### Goal
Every live order attempt must have a unique, stable identifier that:
- is generated by the backend
- is sent to the broker
- can later be used for reconciliation

#### Expected result
The system can relate:
- idempotency_key
- local intent
- client_order_id
- broker-side order lookup

#### Closure criteria
- client_order_id generation implemented
- propagation to broker implemented
- local association recorded
- minimal test coverage added
- validation executed

---

### 1.2 Local attempt traceability

#### Goal
The backend must be able to reconstruct what happened to a live execution attempt.

#### Minimum associated data
- user
- symbol
- side
- idempotency_key
- client_order_id
- broker response / lookup state
- timestamp

#### Expected result
The backend can answer:
- was the order sent?
- which client_order_id was used?
- what is the current known broker-side state?

#### Closure criteria
- traceability fields exist in the relevant backend flow
- the relationship between intent and broker-facing identifier is observable
- minimal verification exists

---

### 1.3 Reconciliation on error/timeout

#### Problem
A local timeout or transport error does not always mean the broker did not receive or execute the order.

#### Goal
When the live path fails with technical uncertainty, the backend must be able to verify real broker-side state when applicable.

#### Expected result
The system can distinguish between:
- NOT_SENT
- SENT_UNKNOWN
- EXECUTED
- FAILED

#### Closure criteria
- reconciliation decision path exists
- technical error vs remote execution state is no longer blindly conflated
- targeted validation exists

---

### 1.4 Clear operational states

#### Goal
Reduce ambiguity in backend responses and execution status tracking.

#### Target examples
- success
- failed
- pending_verification
- reconciled

#### Closure criteria
- operational states are explicitly represented in the backend flow
- behavior is testable and observable

---

## Stage 2 — Robustness Coverage

### Goal

Validate the backend robustness added in Stage 1.

### Planned sub-stages

#### 2.1 client_order_id propagation test
- verify generation
- verify propagation
- verify traceability

#### 2.2 timeout reconciliation test
- simulate local timeout after dispatch
- verify reconciliation behavior

#### 2.3 remote rejection classification test
- simulate broker-side rejection
- verify correct classification

#### 2.4 concurrency plus reconciliation test
- same key or same logical intent under pressure
- verify no double execution
- verify post-error state handling remains correct

---

## Stage 3 — Backend Structural Simplification

### Goal

Improve maintainability without changing product behavior.

### Planned work
- extract Binance auto-pick live pipeline into a clearer local internal unit
- clarify runtime seams
- reduce complexity concentration in ops.py

### Constraint
This stage is not for behavior change.
It is a structure-only stage after robustness has been completed.

---

## Stage 4 — Frontend Review

### Goal

Ensure frontend reflects the backend’s execution states correctly.

### Planned work
- review execution status mapping
- review reconciliation-related states
- review UX for uncertain / pending verification cases
- align frontend expectations with backend operational states

### Constraint
Frontend remains visible in the roadmap, but it is not mixed into the current backend robustness stage.

---

## Active Rule for Project Execution

From this roadmap onward:

- each iteration must produce functional progress
- documentation alone is not enough
- notes/backlog-only loops are not acceptable as a substitute for delivery
- exceptions to the normal workflow are allowed only when they unlock a specific stage and are then closed immediately
